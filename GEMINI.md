## **1. 구현 작업 원칙 — SOLID & Clean Architecture**

### **기본 설계 원칙**

* **SRP**: 단일 책임 원칙
* **OCP**: 개방-폐쇄 원칙
* **LSP**: 리스코프 치환 원칙
* **ISP**: 인터페이스 분리 원칙
* **DIP**: 의존성 역전 원칙

### **개발 방식**

* **TDD 우선**

  * 테스트 작성 → 구현
* **Clean Architecture 적용**

  * 관심사 분리
  * 계층 간 결합 최소화
* **작업 전**

  * Sequential Thinking MCP로 논리적 절차 수립

## **16. 문서 규칙 (Documentation Rules)**

### **16-1. 문서 기본 원칙**

* 문서는 코드의 계약 (Contract)
* 코드 ↔ 문서 불일치 ❌
* 문서 없는 기능 = 미완성

### **16-2. 문서 필수 세트**

* **README.md**

  * 프로젝트 개요
* **ARCHITECTURE.md**

  * 구조 / 계층 / 의존성
* **RUNNING.md**

  * 실행 및 테스트
* **TroubleShoot.md**

  * 장애·디버깅 이력
* **/docs/*.md**

  * 컴포넌트 설명

### **16-3. 문서 작성 규칙**

* 결론 → 근거 → 세부
* 장황한 서술 ❌
* 표 / 체크리스트 중심

### **16-4. 문서 동기화 트리거**

* 구조 변경
* public interface 변경
* I/O 계약 변경
* 실행 방식 변경

  * → 문서 수정 필수

### **16-5. 문서 스타일**

* 언어: 한국어
* 기술 용어: 원문
* 감정 / 추측 ❌

## **17. RUNNING.md 생성 규칙 (강제)**

### **17-1. 목적**

* 사용자가 즉시 실행·테스트 가능해야 함

### **17-2. RUNNING.md 필수 템플릿**

```md
# RUNNING.md

## 1. 실행 개요
## 2. 필수 프로그램
## 3. 환경 변수 설정
## 4. 실행 방법 (로컬)
## 5. Docker 실행
## 6. 정상 동작 확인
## 7. 테스트 실행
## 8. 자주 발생하는 오류
```

### **17-3. 작성 원칙**

* 복사 → 붙여넣기 → 실행 가능
* 추측 표현 ❌
* 실행 불가 = 문서 오류
